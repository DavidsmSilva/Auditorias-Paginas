Pipeline

Flujo de Auditoría Automática

Cuando ejecutás ./audit.sh --audit https://ejemplo.com, el sistema orquesta este pipeline completo:

1
Normalización de URL
Agrega scheme si falta, extrae dominio, hostname, y base URL
2
Pre-flight DNS + Reachability
Resuelve DNS (con 4 métodos de fallback), detecta WAF/CDN, verifica reachability HTTP, crea estructura de directorios
3
Fase 1: Assessment
WhatWeb · Nmap + NSE · Nikto · SSL/TLS · DNS · Nuclei · WPScan · Gobuster
4
Fase 2: Malware Analysis
Descarga de recursos · ExifTool · YARA · Análisis de cabeceras de seguridad HTTP
5
Fase 3: Brand Protection
dnstwist · theHarvester · Sublist3r · Have I Been Pwned
6
Fase 4: IR Readiness
Checklist de preparación ante incidentes (tcpdump, logging, backup, forense, memoria, integridad)
7
Fase 5: SAST
Semgrep · TruffleHog · Gitleaks · Bandit
8
Fase 6: SCA + SBOM
Trivy · Dependency-Check · Syft → SBOM · Grype → Vulns · OSV-Scanner
9
Generación de Reportes
Consolidación de hallazgos → TXT · JSON · HTML · DOCX · PDF

Configuración

Tools DB & Settings

El corazón del sistema es tools.db, un registro centralizado que define cada herramienta con sus metadatos: fase, paquete, binario, método de instalación, descripción y dependencias.

TOOL_PHASE[nmap]="Assessment"
TOOL_METHOD[nuclei]="go"
TOOL_PKG[trivy]="trivy"
TOOL_DEPS[semgrep]="python3"
TOOL_BIN[testssl]="testssl.sh"
TOOL_DESC[nmap]="Escáner de puertos..."
7 arrays asociativos = 78 herramientas con metadatos completos
⚙️ settings.cfg
Configuración del usuario: log level, timeouts, auto-install de dependencias, skip de tools instaladas, formato de reportes, mirror APT personalizado.
config/settings.cfg
🗂️ tools.db
Registro maestro con 7 arrays: TOOL_PHASE, TOOL_PKG, TOOL_DESC, TOOL_BIN, TOOL_METHOD, TOOL_DEPS. Cada herramienta tiene entrada en todos los arrays.
config/tools.db (512 líneas)
Seguridad

Supply Chain Hardening

PaginasAudit implementa múltiples capas de seguridad en la cadena de suministro, siguiendo la regla de Security Due Diligence pre-instalación.

🔒 npm Hardening Global

ignore-scripts=true bloquea lifecycle hooks maliciosos. audit-signatures verifica proveniencia. package-lock=true fija versiones.

🔗 Hash Pinning (SRI)

Antes de instalar un paquete npm, obtiene el hash de integridad del registro y lo verifica post-instalación. Si no coincide → alerta inmediata.

📌 Version Pinning

Trivy pinneado a v0.69.3 (CVE-2026-33634 en v0.69.4).
Syft >= 1.42.3 (CVE-2026-33481).
Semgrep >= 1.157.0 (CVE-2026-34073).

🔍 npm Audit Verification

Verifica el origen del paquete npm via apt-cache policy. Si no viene del repositorio oficial de Debian/Kali → warning.

🧪 Vulnerability Audit

Post-instalación corre npm audit --audit-level=high y reporta vulnerabilidades. Fallos en npm audit signatures elevan alertas.

🛡️ Deb packaging verification

Trivy se instala desde GitHub release DEB con verificación explícita del archivo descargado antes de instalarlo con dpkg.

Build & Deploy

Delivery del Producto

El proyecto soporta 3 modos de ejecución: código fuente, bundle autocontenido e instalador one-command.

📁
Modo Fuente
Clonás el repo y ejecutás bash paginas-auditorias/audit.sh.
Ideal para desarrollo, debug y contribuciones.
📦
Bundle Autocontenido
build.sh genera paginas-audit (7,044 líneas), un ejecutable único sin dependencias externas de archivos. Se deploya con installer.sh a /usr/local/bin/paginas-audit.
Ideal para producción: un solo binario, cero dependencias de carpetas.
🚀
One-Command Install
curl -sL https://raw.githubusercontent.com/DavidsmSilva/Auditorias-Paginas/main/installer.sh | bash
Clona el repo, construye el bundle, lo instala en el sistema y abre el menú interactivo.
Ideal para usuarios finales: un comando y ya está.
